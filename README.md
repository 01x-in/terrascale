# TerraScale — Multi-Tenant Lifecycle Manager for Terraform

## A CLI tool that sits on top of any Terraform codebase and manages tenant provisioning, isolation, and lifecycle.

---

## 1. The Problem

Every engineering team running Terraform faces the same inflection point: the infrastructure works for one tenant. Now make it work for N tenants — without copy-pasting `.tf` files, manually managing state files, or building bespoke scripts that only one person understands.

The current landscape has gaps:

- **Terraform Workspaces** isolate state but don't handle tenant provisioning logic, registry, or lifecycle
- **Terragrunt** keeps configs DRY but isn't tenant-aware — it's a general-purpose wrapper, not a tenancy tool
- **Terraform Cloud / Spacelift** are full platforms with pricing, lock-in, and opinions about your entire workflow
- **`for_each` patterns** are a technique, not a product — every team reinvents the wheel

**TerraScale fills the gap**: a lightweight, opinionated CLI that sits on top of *any* existing Terraform codebase and handles tenant lifecycle — provisioning, inspection, updates, and decommissioning — with isolated state per tenant.

### Mental Model

```
docker-compose : Docker :: terrascale : Terraform (for multi-tenancy)
```

It doesn't replace Terraform. It orchestrates it for the tenant use case.

### Three Layers

**Layer 1: CLI Interface** — Developer-facing commands. Six core commands for the full tenant lifecycle.

**Layer 2: Tenant Registry** — A YAML manifest (`terrascale.yaml`) that tracks every tenant: slug, tier, configuration variables, provisioning status, timestamps, and Terraform output references.

**Layer 3: Terraform Orchestration Engine** — Generates per-tenant `.tfvars`, manages state isolation (workspace-based or directory-based), executes `terraform init/plan/apply/destroy` scoped to a single tenant, and captures outputs back into the registry.

## 3. Core Concepts

### 3.1 Tenant Spec

When you run `terrascale init`, the tool asks you to define a **tenant spec** — a template that describes what differs per tenant. This is stored in `terrascale.yaml`:

```yaml
# terrascale.yaml — generated by `terrascale init`
version: "1"

project:
  name: "oncology-emr"
  terraform_dir: "."              # root of your TF project
  module: "modules/tenant-infra"  # which module = one tenant
  
state:
  backend: "local"                # "local" | "s3"
  # S3 backend config (when backend: s3)
  # s3_bucket: "mycompany-tf-state"
  # s3_region: "us-east-1"
  # dynamodb_table: "tf-locks"

tenant_spec:
  # Variables that differ per tenant
  # Maps to Terraform variables in your module
  variables:
    - name: "tenant_name"
      type: "string"
      required: true
      prompt: "Hospital / Organization name"
    - name: "tenant_slug"
      type: "string"
      required: true
      prompt: "URL-safe identifier (e.g., city-hospital)"
      validation: "^[a-z0-9-]+$"
    - name: "tier"
      type: "string"
      required: false
      default: "standard"
      prompt: "Tenant tier"
      options: ["basic", "standard", "premium"]
    - name: "db_instance_class"
      type: "string"
      required: false
      default: "db.t3.micro"
    - name: "region"
      type: "string"
      required: false
      default: "us-east-1"

  # Terraform outputs to capture per tenant
  outputs:
    - "db_endpoint"
    - "db_name"
    - "s3_bucket"
    - "tenant_api_key"

# Managed by terrascale — do not edit manually
tenants: []
```

### 3.2 Tenant Record

Each provisioned tenant is tracked in the `tenants` array:

```yaml
tenants:
  - slug: "city-hospital"
    name: "City General Hospital"
    tier: "premium"
    status: "active"           # provisioning | active | updating | destroying | destroyed | failed
    created_at: "2026-03-07T10:30:00Z"
    updated_at: "2026-03-07T10:35:00Z"
    variables:
      tenant_name: "City General Hospital"
      tenant_slug: "city-hospital"
      tier: "premium"
      db_instance_class: "db.t3.medium"
      region: "us-east-1"
    outputs:
      db_endpoint: "city-hospital.cluster-xyz.us-east-1.rds.amazonaws.com"
      db_name: "city_hospital_db"
      s3_bucket: "oncology-emr-city-hospital"
      tenant_api_key: "tsk_abc123..."
    state_path: ".terrascale/state/city-hospital/"
```

### 3.3 State Isolation Strategy

TerraScale supports two state isolation modes:

**Directory-based (default, recommended):** Each tenant gets a separate state directory under `.terrascale/state/{tenant-slug}/`. This is the safest — one tenant's `terraform destroy` can never accidentally touch another tenant.

```
.terrascale/
  state/
    city-hospital/
      terraform.tfstate
    memorial-medical/
      terraform.tfstate
    county-clinic/
      terraform.tfstate
```

**Workspace-based:** Uses Terraform workspaces. Simpler but all state lives in one backend — riskier for destructive operations.

**Remote (S3 + DynamoDB):** For team usage. Each tenant gets a unique state key:

```
s3://mycompany-tf-state/terrascale/city-hospital/terraform.tfstate
s3://mycompany-tf-state/terrascale/memorial-medical/terraform.tfstate
```

## 4. CLI Commands — Full Specification

### 4.1 `terrascale init`

**Purpose:** Initialize TerraScale in an existing Terraform project.

**Behavior:**
1. Scans current directory for `.tf` files and `modules/` directory
2. Lists discovered modules, asks which one represents a "tenant unit"
3. Scans the selected module's `variables.tf` to auto-discover variable definitions
4. Prompts user to mark which variables are tenant-specific vs. shared
5. Asks for state backend preference (local / S3)
6. Generates `terrascale.yaml` and creates `.terrascale/` directory
7. Adds `.terrascale/state/` to `.gitignore`

```bash
$ terrascale init

 Scanning Terraform project...
 Found modules: networking, database, compute, tenant-infra, monitoring

? Which module represents a single tenant? tenant-infra

 Discovered 8 variables in modules/tenant-infra/variables.tf

? Select tenant-specific variables (space to toggle):
  [x] tenant_name (string)
  [x] tenant_slug (string)
  [x] tier (string, default: "standard")
  [x] db_instance_class (string, default: "db.t3.micro")
  [ ] vpc_id (string)           ← shared, same for all tenants
  [ ] ecs_cluster_arn (string)  ← shared, same for all tenants
  [x] region (string, default: "us-east-1")
  [ ] environment (string)      ← shared

? State backend: local

 Created terrascale.yaml
 Created .terrascale/ directory
 Added .terrascale/state/ to .gitignore
 Ready! Run `terrascale add` to provision your first tenant.
```

**Flags:**
- `--module <path>` — Skip interactive module selection
- `--backend <local|s3>` — Skip backend prompt
- `--non-interactive` — Use defaults for all prompts

---

### 4.2 `terrascale add <slug>`

**Purpose:** Provision a new tenant.

**Behavior:**
1. Validates slug format (lowercase alphanumeric + hyphens)
2. Checks slug doesn't already exist in registry
3. Prompts for required variables (or accepts via flags)
4. Creates tenant state directory
5. Generates per-tenant `.tfvars` file
6. Runs `terraform init` → `terraform plan` → prompts for confirmation → `terraform apply`
7. Captures outputs and stores in registry
8. Updates tenant status: `provisioning` → `active` (or `failed`)

```bash
$ terrascale add city-hospital

 Creating tenant: city-hospital

? Hospital / Organization name: City General Hospital
? Tenant tier [standard]: premium
? DB instance class [db.t3.micro]: db.t3.medium
? Region [us-east-1]: us-east-1

 Generated .terrascale/state/city-hospital/tenant.tfvars
 Running terraform init...
 Running terraform plan...

Plan: 4 to add, 0 to change, 0 to destroy.

? Apply this plan? [y/N]: y

 Running terraform apply...
 Tenant city-hospital provisioned successfully!

Outputs:
  db_endpoint:  city-hospital.cluster-xyz.rds.amazonaws.com
  db_name:      city_hospital_db
  s3_bucket:    oncology-emr-city-hospital

 Registry updated. Run `terrascale list` to see all tenants.
```

**Flags:**
- `--var tenant_name="City Hospital"` — Set variables inline
- `--tier premium` — Shorthand for tier variable
- `--auto-approve` — Skip plan confirmation (CI/CD usage)
- `--dry-run` — Generate tfvars and show plan without applying

**Flags:**
- `--var key=value` — Update specific variables
- `--tier <tier>` — Change tenant tier
- `--auto-approve` — Skip confirmation

**Purpose:** Decommission a tenant and destroy its infrastructure.

```bash

  - aws_db_instance.tenant
  - aws_s3_bucket.tenant
```

**Flags:**
- `--auto-approve` — Skip confirmation (CI/CD)
- `--keep-state` — Don't delete state files (for auditing)

## 7. Terraform Orchestration — How It Works Under The Hood

### 7.1 What happens during `terrascale add city-hospital`

```
Step 1: Validate
  └── Check slug format, check uniqueness in registry

Step 2: Collect Variables
  └── Prompt for each required variable in tenant_spec
  └── Apply defaults for optional variables

Step 3: Create State Directory
  └── mkdir -p .terrascale/state/city-hospital/

Step 4: Generate tfvars
  └── Write .terrascale/state/city-hospital/tenant.tfvars
  └── Contents:
      tenant_name      = "City General Hospital"
      tenant_slug      = "city-hospital"
      tier             = "premium"
      db_instance_class = "db.t3.medium"
      region           = "us-east-1"

Step 5: Write backend override
  └── Write .terrascale/state/city-hospital/backend.tf
  └── Points state to this tenant's directory
  └── (For S3: unique key per tenant)

Step 6: Terraform Init
  └── Run: terraform init \
        -backend-config="path=.terrascale/state/city-hospital/terraform.tfstate" \
        -chdir=modules/tenant-infra
  └── (Or copy module to temp dir with backend override)

Step 7: Terraform Plan
  └── Run: terraform plan \
        -var-file=.terrascale/state/city-hospital/tenant.tfvars \
        -out=.terrascale/state/city-hospital/plan.out
  └── Show plan to user

Step 8: Confirm & Apply
  └── Run: terraform apply .terrascale/state/city-hospital/plan.out

Step 9: Capture Outputs
  └── Run: terraform output -json
  └── Parse and store in registry

Step 10: Update Registry
  └── Add tenant record to terrascale.yaml
  └── Set status: active
```

### 7.2 State Isolation Detail

The critical challenge is running the same module multiple times with different state. TerraScale handles this by creating a **working directory per tenant** that contains:

```
.terrascale/state/city-hospital/
├── terraform.tfstate       # This tenant's state
├── tenant.tfvars           # This tenant's variables
├── backend_override.tf     # Backend config pointing here
└── plan.out                # Last plan (temporary)
```

The executor copies (or symlinks) the module source into a temporary execution context, applies the backend override, and runs Terraform there. This ensures complete isolation — no tenant can ever see or affect another tenant's state.

### 7.3 Working with Existing Terraform Projects

TerraScale doesn't require restructuring your Terraform code. It works with your modules as-is. The only requirement:

- You have a Terraform module that represents "one tenant's infrastructure"
- That module accepts variables for tenant-specific configuration
- That module outputs the resources created (endpoints, names, etc.)

If your current project is a flat structure (no modules), TerraScale's `init` will suggest wrapping your resources in a module first.

## 12. Demo Module (For Testing)

Create a simple Terraform module that TerraScale can manage. This module doesn't need real AWS resources for testing — use `local_file` and `null_resource` to simulate:

```hcl
# demo/modules/tenant/variables.tf

variable "tenant_name" {
  type        = string
  description = "Name of the tenant organization"
}

variable "tenant_slug" {
  type        = string
  description = "URL-safe tenant identifier"
}

variable "tier" {
  type        = string
  default     = "standard"
  description = "Tenant tier: basic, standard, premium"
}

variable "region" {
  type        = string
  default     = "us-east-1"
  description = "AWS region for tenant resources"
}
```

```hcl
# demo/modules/tenant/main.tf

resource "local_file" "tenant_config" {
  content = jsonencode({
    tenant_name = var.tenant_name
    tenant_slug = var.tenant_slug
    tier        = var.tier
    region      = var.region
    provisioned = timestamp()
  })
  filename = "${path.module}/../../.tenant-data/${var.tenant_slug}/config.json"
}

resource "random_password" "api_key" {
  length  = 32
  special = false
}
```

```hcl
# demo/modules/tenant/outputs.tf

output "db_endpoint" {
  value = "${var.tenant_slug}.db.example.com"
}

output "db_name" {
  value = "${replace(var.tenant_slug, "-", "_")}_db"
}

output "s3_bucket" {
  value = "myapp-${var.tenant_slug}-storage"
}

output "tenant_api_key" {
  value     = random_password.api_key.result
  sensitive = true
}
```

This lets you demo the full TerraScale workflow without spinning up real infrastructure or incurring AWS costs.

## 13. Naming Considerations

| Name | Pros | Cons |
|------|------|------|
| **terrascale** | Clear meaning, "terraform + scale" | Might imply scaling, not tenancy |
| **tftenants** | Extremely clear | Sounds generic |
| **tentform** | Unique, memorable | Meaning not immediately obvious |
| **terratenant** | Self-documenting | Long to type |
| **multitf** | Short, punchy | Vague |
| **tenantctl** | Follows k8s naming (kubectl) | Doesn't mention Terraform |

**Recommendation:** Go with **terrascale** — it implies scaling Terraform infrastructure, which is exactly what multi-tenancy is. It's also easy to type, memorable, and available as a GitHub repo name.
